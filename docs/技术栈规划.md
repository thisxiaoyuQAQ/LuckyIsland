# LuckyIsland 当前技术栈与架构基线

> Windows 灵动岛式桌面助手 · 当前实现基线
> 更新时间：2026-07-13

## 0. 决策摘要

| 维度 | 当前实现 |
|---|---|
| 应用框架 | Tauri 2（Rust + WebView2） |
| 前端 | React 19 + TypeScript + Vite 7 |
| 样式/动画 | Tailwind CSS v4 + shadcn 风格组件 + motion 12 |
| 状态 | React 本地状态 + `useSyncExternalStore` 外部 store；不使用 Zustand/TanStack Query |
| 配置/数据 | `app_data_dir()/data.db` 中的 SQLite；应用设置真源为 `settings(key,value)` KV |
| 终端 | `@xterm/xterm` + `portable-pty` |
| 通知 | `lucky-notify` 直接 POST `127.0.0.1:9753/notify`；无命名管道和离线队列 |
| AI | Claude CLI / Codex CLI / 自定义 Chat API；Provider 各自按实现联网；完整结果非流式返回，支持前端 pending 状态与真正取消 |
| 语音 | sherpa-onnx KWS + 流式 ASR、cpal 采集、Windows SAPI TTS |

早期文档中的 Zustand、TanStack Query、应用 `config.toml` 热重载、Porcupine、命名管道通知和 `codex --search` 均未成为当前实现，不应作为现行架构补建。

## 1. 整体架构

```text
React 19 / TypeScript / Vite 7
  ├─ island：单页画布、三态、设置与页面交互
  ├─ ai-palette：独立 AI 窗口
  ├─ React 本地状态
  └─ useSyncExternalStore 外部 store
             ↕ Tauri invoke / event
Rust / Tauri 2
  ├─ monitor + window：多屏、定位、三态、双动作热键
  ├─ data：天气、股票、日历、待办、time_api
  ├─ terminal：portable-pty
  ├─ notify：本地 HTTP、分发、历史、Windows toast
  ├─ ai：provider、router、runtime、history
  ├─ voice：sherpa-onnx KWS/流式 ASR、cpal、SAPI
  └─ storage：SQLite schema 与 settings KV
```

前端负责展示与交互；系统 IO、网络、本地进程、SQLite 和窗口能力由 Rust 承担。跨窗口持久状态最终以 SQLite settings KV 为准。

## 2. 前端基线

事实源：`package.json`、`src/App.tsx` 与实际 `src/` 目录。

- React 19、TypeScript、Vite 7。
- Tailwind CSS v4；动画使用 `motion`。
- 终端使用 `@xterm/xterm`、fit/web-links/webgl addons。
- 当前状态由组件本地状态及项目内外部 store 管理；依赖中没有 Zustand 或 TanStack Query。
- `src/App.tsx` 负责岛窗口页面切换、三态、主题/毛玻璃、通知事件与窗口交互。

## 3. Rust/Tauri 基线

事实源：`src-tauri/Cargo.toml`、`src-tauri/src/lib.rs` 与实际目录。

当前注册并使用的能力以 Cargo 与 `lib.rs` 为准，包括全局快捷键、单实例、通知、自启动、opener、dialog、窗口/托盘，以及项目自己的 SQLite、HTTP、PTY、AI 和语音模块。不得把早期规划中的 `tauri-plugin-store`、`tauri-plugin-os`、`tauri-plugin-shell`、`tauri-plugin-clipboard-manager` 或不存在的 config/window/theme 目录写成已实现插件或模块。

```text
src-tauri/src/
  ai/          provider、router、runtime、history
  bin/         lucky-notify
  data/        weather、stock、calendar、todo、time_api
  notify/      HTTP server 与通知分发
  storage/     SQLite 初始化、settings KV、导入导出
  terminal/    portable-pty 生命周期
  voice/       sherpa-onnx KWS/ASR、cpal、SAPI
  lib.rs       Tauri setup、插件、命令与托盘注册
  monitor.rs   显示器选择、定位与运行时断屏监听
  settings.rs  通用 settings 命令
```

## 4. 配置、存储与日志

- 数据库路径：Tauri `app_data_dir()/data.db`。
- 应用设置：SQLite `settings(key,value)`；前端通过 `setting_get/set` 及 `settings://changed` 同步。
- 窗口设置包括 `window:monitor`、`window:opacity`、`window:offset_x`、`window:offset_y` 以及两个动作热键对应设置。
- 当前 schema 在 Rust 中内联初始化；不能描述为已有版本化 migration 目录。
- 当前错误主要以字符串返回，日志主要为控制台/`eprintln!`；结构化、脱敏、持久轮转日志尚未完成。
- 需求文档中的应用 `config.toml` 示例是早期未采用草案。外部 Codex 的 `~/.codex/config.toml` hook 配置仍合法，与应用配置无关。

## 5. 通知

`lucky-notify` 是独立 CLI，但它本身是本地 HTTP 客户端：直接向 `127.0.0.1:9753/notify` POST JSON，并处理 token/no-proxy。主程序接收后写 SQLite 历史、向前端分发并按设置触发 Windows toast。

当前没有命名管道、离线待发送队列或 `notify/ipc.rs`。主程序离线时 CLI 请求会失败，不承诺排队补发。

## 6. AI 与联网

- Provider 固定为 Claude CLI、Codex CLI、自定义 Chat API 三类。
- AI 回答为一次性完整结果；这里的“非流式”只指 AI token/答案，不影响语音的流式 ASR。
- 前端在请求期间显示 pending/忙碌占位；当前没有后端 AI 分阶段进度事件，不把 pending 描述成可观测搜索进度。
- 前后端以 requestId 维护单活请求，并支持真正取消。CLI 取消终止 Windows 进程树，HTTP/tool 链路使用 cancellation token。
- Codex 实时问题先由 Rust 后端分类：天气走天气源取证，其他实时信息走 DuckDuckGo 预搜索，再把证据注入 Codex prompt；查证失败时不调用 Codex。
- 自定义 Chat API 可通过后端 DuckDuckGo `web_search` function tool 取证；Claude CLI 使用自身允许的 `WebSearch`，不能概括成应用统一预搜索。
- Codex 不依赖 `codex --search`；无足够证据时不得伪装成实时回答。

## 7. 语音

当前链路为：

```text
cpal 麦克风 PCM
  → sherpa-onnx KWS 唤醒
  → sherpa-onnx 流式 ASR（按需懒加载/录音）
  → AI 完整回复
  → Windows SAPI TTS
```

Porcupine/Picovoice/AccessKey 是早期未采用方案，不是当前依赖或配置。必须保留“流式 ASR”的准确表述，但不得据此宣传 AI 回答流式输出。

## 8. 多屏、窗口与热键

- `window:monitor` 保存动态主屏或具体显示器 ID。
- 运行时监听显示器；所选副屏断开时应用主动回退主屏并保留选择。
- 副屏恢复时后端不主动重新定位；曾在 Windows 真机观察到系统自动移回原屏，但这不是稳定产品保证。
- 当前有两个可配置动作热键：岛显示/隐藏与 AI 面板唤起；页面导航另有固定快捷键。
- 点击穿透的最终产品契约留给 DOC-10A-03，本基线不宣称新增行为。
