# LuckyIsland 展开/收起动画优化设计

> 日期：2026-07-14
> 状态：待实现
> 范围：模块 11 / Phase 2 / Task 6 动画打磨补充

## 目标

优化灵动岛展开与收起的视觉编排，采用用户确认的方案 B：**容器先展开，内容随后进入；收起时内容先退出，窗口再缩小**。

本次只调整动画节奏和表现，不改变窗口策略归约、悬停 180ms/300ms 判定、穿透逻辑、设置持久化或模块 10 内容。

## 当前问题

当前 Rust 平台窗口在状态变化时直接切换至 720×80 或 720×400，React 外层容器使用统一高度 transition。窗口尺寸、容器高度和主体内容同时变化，使展开看起来像整体拉伸，收起时内容与窗口一起被压缩，缺少层次。

## 已确认方案

### 展开

1. Rust 窗口先获得展开尺寸，避免 WebView 内容被旧窗口边界裁切。
2. 外层容器从紧凑高度向下展开，时长约 240ms。
3. 使用 `cubic-bezier(0.2, 0.8, 0.2, 1)`，保持快速响应并平滑减速。
4. 主体内容延迟约 60ms 后进入，在约 180ms 内完成：
   - `opacity: 0 → 1`
   - `translateY(-8px) → 0`
5. 顶部栏保持固定，不参与上下位移。

### 收起

1. 主体内容先在约 120ms 内淡出并轻微上移。
2. 内容退出后，外层容器与平台窗口进入紧凑尺寸。
3. 收起总时长约 200ms，快于展开，避免拖沓。
4. 保留当前“先播放前端收缩，再提交后端 Compact”的正确方向，但把延迟改为与退出动画完成点一致。

## 状态与并发安全

- 复用现有 `createIslandTransitionController` 的 generation 机制。
- 任一新请求都会取消旧请求的待执行延迟。
- 展开打断收起时，不得提交过期的 Compact 平台请求。
- 收起打断展开时，不接受旧展开请求返回的快照。
- 悬停控制器仍只负责 180ms/300ms 的 hover 意图；动画控制器负责视觉编排，两者职责不合并。

## React 结构

- 顶部栏继续作为稳定的 `h-14 shrink-0` 区域。
- 主体内容使用独立的 Motion 容器或等价动画层。
- 展开态主体按延迟淡入/上移。
- 收起请求先将主体切换到退出态，退出完成后由 transition controller 提交 Compact。
- 不重新组织页面组件，不修改页面业务状态。

## Reduced Motion

检测 `prefers-reduced-motion: reduce`：

- 取消 `translateY`；
- 取消 60ms 内容延迟；
- 高度和透明度使用极短过渡或直接切换；
- 仍保持状态提交顺序和 generation 取消语义。

## 测试

扩展现有 `src/lib/__tests__/window-policy.test.ts`：

1. 展开请求立即提交平台状态，主体进入由视觉层处理。
2. 收起请求先进入退出态，在退出延迟完成前不提交 Compact。
3. 展开请求可取消待提交的 Compact。
4. 旧请求返回不得覆盖新 generation。
5. reduced-motion 路径不等待内容退出延迟。

验证命令：

```bash
pnpm test src/lib/__tests__/window-policy.test.ts
pnpm exec tsc --noEmit
pnpm build
git diff --check
```

最后由用户在 Windows 真机观察：展开无整体拉伸感、内容不被裁切、顶部栏不漂移、收起无压扁感、快速反向操作无闪烁。
