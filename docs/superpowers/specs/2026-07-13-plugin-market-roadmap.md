# LuckyIsland 插件系统与公开插件市场路线图

> 日期：2026-07-13
> 状态：路线规划已获用户批准；阶段 0 已于 2026-07-14 完成；阶段 1 尚未立项；禁止直接作为实施计划执行
> 长期目标：支持公开第三方插件生态，并把语音、问答迁移为可真正安装/卸载的官方插件
> 当前边界：阶段 0 只完成威胁模型与产品规则；不实现插件运行时、市场服务、模块迁移或卸载入口

## 1. 背景与决策

用户希望“语音和问答模块支持卸载”，并明确后续要增加插件系统与公开插件市场。简单隐藏入口或删除模型不能满足“真正卸载”：

- 语音运行时 DLL 仍随主安装包加载；
- Rust 模块仍编译在主程序；
- 问答 Provider、窗口、快捷键和历史仍与宿主耦合；
- 无法由第三方以相同机制安装、更新和卸载能力。

因此，长期方案不是在现有设置页增加删除按钮，而是建设受控插件平台，并把语音、问答作为首批官方插件迁移。该项目涉及执行不可信代码、原生能力、市场审核与供应链安全，必须分阶段另行设计和实施。

## 2. 目标

长期必须达到：

- 插件可安装、启用、禁用、更新、回滚、保留数据卸载和彻底卸载；
- 第三方插件崩溃或卡死不带崩 LuckyIsland 主进程；
- 插件按 manifest 声明能力，默认拒绝未授权访问；
- 插件 UI、后台任务、快捷键、托盘、通知和 AI Provider 通过版本化扩展协议接入；
- 市场提供开发者身份、签名、审核、发布、更新、下架、撤销和安全公告；
- 语音插件卸载后可释放 sherpa-onnx DLL、模型和缓存；
- 问答插件卸载后主程序仍完整可用；
- 市场不可用时，已安装且未被撤销的插件可继续运行；
- 插件协议与宿主版本兼容性明确、可测试、可回滚。

## 3. 非目标与安全底线

路线图不承诺：

- “审核通过”等于绝对安全；
- 在没有 OS 沙箱/进程边界时允许第三方原生 DLL 进入主进程；
- 插件可任意调用 Tauri command、控制窗口或读取主数据库；
- 第一版同时支持 Windows、macOS、Linux；
- 第一版实现账号计费、付费插件、分成和 DRM；
- 市场紧急下架时静默删除用户数据。

安全底线：

- 第三方二进制不得动态加载进 LuckyIsland 主进程；
- 插件 UI 不获得宿主全局 Tauri `invoke`；
- 默认拒绝未声明、未授权能力；
- 市场索引、插件包和更新都必须可验证；
- 插件不能读取其他插件数据；
- 用户主动隐藏/全屏/焦点等窗口政策仍由宿主裁决；
- 插件停止或卸载后必须释放进程、麦克风、快捷键、文件句柄和订阅。

## 4. 选定架构

采用 **受控进程外运行 + 声明式 UI 扩展**：

```text
LuckyIsland 主程序
├── Plugin Manager
│   ├── 安装 / 启用 / 禁用 / 卸载
│   ├── 更新 / 健康检查 / 回滚
│   ├── 签名 / 哈希 / 兼容性校验
│   └── 权限授权与撤销
├── Plugin Bridge
│   ├── 版本化 IPC
│   ├── 能力调用
│   ├── 事件订阅
│   ├── 超时 / 限流 / 配额
│   └── 审计记录
├── UI Extension Host
│   ├── 灵动岛页面
│   ├── 设置页面
│   ├── 快捷键动作
│   ├── 托盘动作
│   └── 状态卡片
└── Plugin Host 进程
    └── 每插件独立进程，或按等价信任域严格分组
```

优点：

- 插件崩溃与主进程隔离；
- 可强制结束超时/恶意插件；
- 原生 DLL 不锁在宿主进程，允许真正卸载；
- 避免 Rust ABI、编译器和 DLL 版本耦合；
- 权限检查集中在 Bridge；
- CPU、内存、请求频率和崩溃次数可观测。

第一版以每插件独立进程为安全基线。只有经过威胁模型和性能数据证明后，才讨论共享 Host；不能为节省少量内存默认把不相干第三方插件混进同一进程。

## 5. 插件类型策略

在专项设计阶段需要在以下执行载体中选择或组合：

1. **进程外原生插件**：适合语音、PTY、CLI Provider 等原生能力；隔离成本高但能力完整。
2. **WASM/WASI 插件**：适合纯计算、网络和结构化数据转换；沙箱更明确，但桌面原生能力需要 Bridge。
3. **隔离 Web UI 扩展**：只负责渲染与用户交互，不承担受信任后端能力。

推荐长期组合：

- UI 始终隔离并通过消息桥；
- 普通数据/自动化插件优先 WASM；
- 只有确需麦克风、原生模型或 CLI 子进程的插件使用独立原生 Host；
- 两类后端共享同一能力模型和协议，不向用户暴露完全不同的权限语义。

具体运行时（WASI 实现、RPC 编码、进程启动约束）必须在阶段 1 的威胁模型和原型后决定，本路线图不提前锁死库。

## 6. 插件包与 Manifest

建议归档格式：

```text
example.luckyplugin
├── manifest.json
├── checksums.json
├── signature.ed25519
├── backend/
│   ├── windows-x86_64/
│   └── wasm/
├── frontend/
├── assets/
├── locales/
└── licenses/
```

Manifest 最小结构：

```json
{
  "schemaVersion": 1,
  "id": "com.example.weather-plus",
  "name": "Weather Plus",
  "version": "1.0.0",
  "publisher": "example",
  "minHostVersion": "0.4.0",
  "maxHostVersion": null,
  "platforms": ["windows-x86_64"],
  "entrypoints": {
    "backend": "backend/windows-x86_64/plugin-host.exe",
    "settings": "frontend/settings.html"
  },
  "contributes": {
    "islandPages": [],
    "settingsPages": [],
    "hotkeyActions": [],
    "trayActions": [],
    "notificationSources": [],
    "aiProviders": []
  },
  "permissions": [],
  "dependencies": [],
  "conflicts": []
}
```

规则：

- 插件 ID 全局唯一，发布后不可更改或复用；
- `schemaVersion` 管包格式，Bridge/API 另有协议版本；
- `minHostVersion/maxHostVersion` 决定宿主兼容范围；
- 平台和架构必须显式声明；
- 所有文件列入哈希清单；签名覆盖 manifest 与完整清单；
- 包内路径规范化，拒绝绝对路径、`..`、符号链接逃逸和同名大小写碰撞；
- 解压前限制总大小、文件数、单文件大小和压缩比，防 zip bomb；
- 插件程序按 `plugin-id/version` 只读存放；插件数据另存稳定数据目录；
- 当前激活版本由数据库/原子指针记录，便于更新和回滚；
- licenses 与第三方声明是发布必填内容。

## 7. 版本化协议

Plugin Bridge 需要明确握手：

```text
Host → Plugin: hostVersion, protocolVersions, grantedCapabilities, locale, theme
Plugin → Host: pluginId, pluginVersion, selectedProtocol, requestedContributions
```

要求：

- 所有请求有 request ID、deadline 和结构化错误；
- 事件订阅有稳定 ID，可主动取消；
- 插件进程重启后旧 request/event identity 失效；
- 未知字段遵循明确向前兼容规则；
- 不允许插件通过自由字符串调用任意宿主方法；
- 每项 capability 有独立 schema、版本和参数限制；
- 大文件/音频流不直接塞进无界 JSON 消息，使用受控流或临时句柄；
- Host 对每插件限制并发数、消息大小、频率和总带宽；
- 审计日志只记录能力、时间、结果和非敏感元数据，不记录 API Key、音频正文或对话全文。

## 8. 权限模型

建议能力：

```text
storage.plugin             访问插件私有数据
storage.user_selected      访问用户明确选择的文件/目录
network.domain             访问声明的 HTTPS 域名
microphone.capture         使用麦克风
notification.publish       发布 LuckyIsland 通知
island.page                注册灵动岛页面
settings.page              注册设置页
hotkey.action              声明由宿主注册的快捷键动作
tray.action                声明托盘动作
process.spawn              启动明确声明的外部程序
clipboard.read
clipboard.write
ai.provider                提供问答 Provider
ai.conversation            发起宿主协调的对话能力
speech.transcribe
speech.synthesize
```

授权规则：

- manifest 预声明，安装时展示；首次使用高风险能力时可再次确认；
- 权限按插件 ID + publisher + major version 绑定；发布者或高风险权限变化需要重新授权；
- 网络授权按 HTTPS 域名/端口白名单，不等于任意联网；重定向后域名也必须获准；
- 文件访问优先使用用户选择句柄，不开放任意路径；
- `process.spawn` 只允许声明的可执行目标和参数模板，禁止任意 shell 字符串；
- 麦克风、剪贴板读取、进程启动属于高风险权限；
- 用户撤销后 Bridge 立即拒绝新调用，并主动终止可撤销的长期会话；
- 插件不能直接读取宿主 SQLite、设置表、通知 Token 或其他插件目录；
- 插件不能直接调用窗口 show/hide/focus/click-through，只能提交受限展示意图；最终由 Window Policy 裁决。

仅有进程边界并不自动形成强 OS 沙箱。Windows 下若要对恶意原生插件提供更强隔离，专项安全设计必须评估 AppContainer、受限 Token、Job Object、文件 ACL、网络代理等机制。达不到可声明的隔离级别时，产品文案必须如实称为“权限代理与进程隔离”，不能宣称完整沙箱。

## 9. UI 扩展

第一阶段不允许插件把任意 JavaScript 模块注入宿主 React 树。插件 UI 运行在隔离 WebView/iframe 风格宿主，只通过 Plugin Bridge 通信。

支持扩展点：

- 灵动岛紧凑摘要与展开页面；
- 设置页面；
- 快捷键动作；
- 托盘动作；
- 通知来源；
- AI Provider；
- 后台服务状态卡片。

插件声明：

- 紧凑/展开入口；
- 支持的最小/建议布局；
- 是否需要键盘输入；
- 空闲时 UI 是否可回收；
- 后台刷新需求；
- 可访问的主题 Token。

宿主控制：

- 页面启用、排序和可见性；
- 主题、字号、动画和减少动态效果；
- 窗口尺寸、全屏抑制、点击穿透；
- 通知优先级与焦点；
- CPU/内存配额和异常处置；
- UI CSP、导航和外链策略。

插件不能修改宿主全局 CSS，也不能依赖 DOM 结构选择器侵入其他页面。设计系统 API 必须版本化。

## 10. 安装流程

```text
下载临时文件
  → 验证市场索引签名与下载 URL
  → 校验包哈希/签名
  → 安全解析归档
  → 校验 manifest / publisher / 平台 / 宿主兼容
  → 静态扫描与风险提示
  → 展示权限和依赖
  → 解压到新版本 staging 目录
  → 启动受限健康检查
  → 原子登记并启用
  → 清理 staging
```

任何一步失败：

- 不修改当前激活版本；
- 清理未激活目录；
- 保留结构化错误供用户/开发者诊断；
- 不执行包内卸载脚本或任意 post-install shell。

依赖必须是能力或稳定插件 ID + 版本范围；安装器先解析完整依赖图、冲突与循环，再请求用户确认。第一阶段可只允许官方插件依赖，降低供应链放大风险。

## 11. 卸载与数据生命周期

用户必须看到两个明确选项：

### 11.1 保留数据卸载

- 停止插件并注销所有扩展；
- 删除程序、运行时、模型和可再生缓存；
- 保留设置、密钥和业务数据；
- 重装同一可信插件时可恢复。

### 11.2 彻底卸载

- 在保留数据卸载基础上删除设置、密钥、业务数据、日志和授权；
- 展示预计删除项与大小；
- 必要时要求二次确认；
- 不删除插件导出到用户选择目录中的文件。

卸载顺序：

1. 标记 stopping，拒绝新调用；
2. 取消后台任务和长期请求；
3. 移除 UI、快捷键、托盘、通知源和事件订阅；
4. 请求进程正常退出；
5. 有界等待，超时后终止进程树；
6. 释放文件/麦克风/模型句柄；
7. 删除程序与选定数据；
8. Windows 文件仍被占用时写入宿主重启后的待清理队列；
9. 重启后验证清理结果并向用户报告残留。

插件不能提供任意卸载脚本。必要的数据迁移/清理通过受限版本化接口完成。

## 12. 更新、迁移与回滚

- 新版本安装到独立目录，绝不覆盖当前版本；
- 新版本签名、兼容性和健康检查通过后才原子切换；
- 保留至少一个最近可用版本，受磁盘策略约束；
- 插件数据 schema 迁移必须声明 from/to 版本和回滚能力；
- 不可逆迁移前创建备份，并明确告知用户；
- 启动崩溃/健康检查失败自动回滚；
- 运行期连续崩溃或资源超限达到阈值时自动禁用，提示诊断和回滚；
- 市场下架阻止新安装，不能静默删除用户数据；
- 确认恶意或密钥泄露时通过签名撤销列表阻止启动，并展示原因、公告和数据导出/删除选项。

更新策略不得只信任版本号；还要绑定 publisher identity。插件 ID 相同但发布者签名变化必须走显式所有权迁移或拒绝。

## 13. 公开插件市场

### 13.1 组成

```text
开发者门户
  → 身份/组织、发布者密钥、插件 ID、版本提交、审核状态

审核流水线
  → 结构校验、签名、恶意软件、依赖/SBOM、权限差异、动态测试、人工审核

用户市场
  → 搜索、分类、版本、开发者、权限、兼容性、更新、评分、举报、安全公告
```

### 13.2 最小 API

```text
GET /v1/index
GET /v1/plugins/{id}
GET /v1/plugins/{id}/versions
GET /v1/plugins/{id}/versions/{version}
GET /v1/plugins/{id}/download/{version}/{platform}
GET /v1/revocations
GET /v1/advisories
```

索引和撤销列表必须可离线验证。下载 URL 使用不可变版本与内容哈希，避免同 URL 内容被替换。

### 13.3 发布者和审核签名

建议分层：

- 发布者签名证明版本由对应开发者/组织发布；
- 市场签名证明该包经过对应审核策略并进入索引；
- 客户端验证两者与插件 ID、版本、hash 的绑定；
- 发布者密钥轮换、组织成员变更和账户恢复要有审计与冷静期；
- 市场签名根密钥离线保护，在线使用中间签名密钥并支持轮换。

### 13.4 审核

自动审核至少包括：

- 包格式、路径、签名和哈希；
- 已知恶意软件和依赖漏洞；
- SBOM 与许可证；
- manifest 权限与二进制行为差异；
- 网络域名、进程执行和原生依赖；
- UI 导航/CSP；
- 安装、启动、禁用、升级、回滚、卸载测试；
- CPU、内存、启动时间和崩溃；
- 权限变化 diff。

高风险能力或混淆二进制进入人工审核。审核策略版本、工具版本和结果摘要可追溯。

### 13.5 市场韧性

- 市场不可用时已安装插件继续工作，除非本地已有有效撤销；
- 索引和撤销列表有签名缓存、有效期和失败降级；
- 不能因市场短时不可达禁用全部插件；
- 离线侧载使用同一包校验，但未获市场签名时显示更强警告；
- 下载支持断点/重试和镜像，但镜像不能改变 hash/signature；
- 安装量、评分和评论需要反滥用，不能由发布者自报。

## 14. 开发者 SDK

在开放市场前提供：

- manifest schema 和生成器；
- Rust/TypeScript/WASM 客户端 SDK（按最终运行时）；
- 本地开发 Host 和模拟能力；
- 权限检查、错误码、主题 Token、UI 组件规范；
- 打包、签名、测试、兼容性检查 CLI；
- 官方最小插件模板；
- 升级/数据迁移模板；
- 发布前本地审计报告。

SDK 必须让插件可以在不读取 LuckyIsland 私有内部代码的情况下理解接口。协议兼容性测试套件作为 SDK 的一部分发布。

## 15. 官方问答插件迁移

迁移内容：

- AI 面板和设置页；
- Claude CLI、Codex CLI、Chat API Provider；
- 对话历史；
- Provider 配置和密钥；
- AI 快捷键动作；
- 请求取消、子进程树和 HTTP/tool 生命周期；
- Provider 状态同步。

宿主最终只保留通用扩展协议，例如：

```text
ai.provider.register
ai.conversation.start
ai.conversation.cancel
ai.conversation.event
```

要求：

- API Key 存插件受保护数据区，不暴露给其他插件；
- CLI Provider 的进程启动必须经过 `process.spawn` 高风险授权和明确命令模板；
- 取消 request identity、late-result 防护和历史写入门禁不能因插件化退化；
- 多个问答插件同时存在时，宿主提供明确默认 Provider 选择，而非插件互相覆盖；
- 卸载问答插件后，主程序仍可显示其他页面和管理插件；
- 保留数据卸载与彻底卸载分别处理历史/密钥。

问答插件应作为第一个复杂官方迁移，用来验证 UI、网络、进程、快捷键、持久化和取消协议。

## 16. 官方语音插件迁移

迁移内容：

- sherpa-onnx 运行时 DLL；
- KWS/ASR 模型下载、校验与缓存；
- 麦克风采集；
- 唤醒词与重载；
- TTS；
- 手动录音；
- 语音到问答的联动；
- 对应设置页和状态事件。

迁移完成后：

- LuckyIsland 主安装包不再包含 sherpa-onnx DLL；
- 安装语音插件时才安装所需运行时，模型继续按需下载；
- 保留数据卸载删除运行时与模型但可保留唤醒词设置；
- 彻底卸载删除运行时、模型、配置、缓存和授权；
- 所有麦克风与模型句柄在卸载前释放；
- Windows loader 不再因宿主启动而要求语音 DLL 存在。

语音与问答通过能力而不是固定插件 ID 解耦：

```text
requires optional: ai.conversation
provides: speech.transcribe, speech.wake, speech.synthesize
```

没有问答能力时，语音插件仍可提供唤醒/转写；需要问答时明确提示安装或选择一个实现 `ai.conversation` 的插件。

由于语音涉及原生 DLL、模型、麦克风和高内存，是验证真正卸载和资源释放的关键阶段，但应排在问答插件与本地插件运行时稳定之后。

## 17. 数据迁移原则

内置模块迁移为插件时不能丢失用户数据：

- 宿主检测旧 AI/voice settings 和历史；
- 安装官方插件后，先复制到插件 staging 数据区；
- 插件验证 schema、密钥可读性和记录数量；
- 用户确认插件工作后才标记宿主旧数据已迁移；
- 在一个兼容周期内保留只读备份或可回滚路径；
- 彻底删除旧数据必须另行确认；
- 密钥迁移不得写入日志或普通导出文件；
- 配置导入导出的插件数据契约由插件声明，宿主不默认导出敏感数据。

## 18. 建设阶段与门禁

### 阶段 0：威胁模型与产品规则

产物：

- 信任边界、攻击面、能力分级；
- 插件类型与支持平台；
- 市场治理、隐私、审核和撤销政策；
- 性能预算；
- 付费能力明确排除或另立项目。

门禁：安全评审批准，不写运行时生产代码。

### 阶段 1：协议与最小原型

- manifest/schema；
- 版本化 IPC；
- 一个无网络、无原生能力的官方示例插件；
- 进程崩溃隔离、启动/停止和基本健康检查。

门禁：宿主不因插件崩溃而退出；未知调用默认拒绝。

### 阶段 2：本地官方插件运行时

- 安装/启用/禁用/更新/回滚/卸载；
- 权限与审计；
- UI extension host；
- 快捷键、设置页、通知等低风险扩展点。

门禁：完整生命周期与故障注入通过，只允许 LuckyIsland 官方签名插件。

### 阶段 3：问答插件迁移

- 迁移全部 Provider、面板、历史、取消和快捷键；
- 旧数据可回滚迁移。

门禁：现有 AI 真机验收矩阵不退化；宿主可在无问答插件时正常运行。

### 阶段 4：语音插件迁移

- 原生 DLL、模型、麦克风、KWS/ASR/TTS；
- 与 `ai.conversation` 能力联动。

门禁：安装后语音链路工作；彻底卸载后 DLL/模型/进程/句柄无残留，主包不再携带运行时。

### 阶段 5：第三方本地侧载

- 发布者签名；
- 强风险提示；
- 高风险权限与 OS 隔离；
- 恶意/异常插件故障注入。

门禁：独立安全审查和外部渗透/红队测试；在此之前不允许任意第三方原生包。

### 阶段 6：开发者 SDK 与审核流水线

- SDK、模板、CLI、兼容测试；
- 自动扫描、SBOM、动态测试和人工审核工具。

门禁：至少两个非核心团队/真实外部试用插件验证文档和协议。

### 阶段 7：封闭市场测试

- 开发者门户、签名索引、下载、更新、举报和安全公告；
- 仅邀请开发者和测试用户。

门禁：密钥轮换、撤销、下架、市场离线和供应链事件演练通过。

### 阶段 8：公开稳定市场

- 公开提交、搜索、评分、更新和治理；
- 稳定 SLA、审核时效、响应流程和透明度报告。

每个阶段必须另写设计规格与实施计划。不能从本路线图直接跳到公开市场编码。

## 19. 测试与安全验证矩阵

### 19.1 生命周期

- 安装中断、磁盘满、损坏包、签名错误；
- 启动超时、崩溃循环、僵尸进程；
- 更新健康检查失败和回滚；
- 卸载时句柄占用、进程拒绝退出、重启后清理；
- 保留数据重装恢复、彻底卸载无残留。

### 19.2 权限

- 未声明能力拒绝；
- 用户拒绝/撤销后立即生效；
- 域名重定向绕过、DNS/URL 规范化；
- 路径遍历、符号链接、大小写碰撞；
- 任意 shell 参数、可执行文件替换；
- 插件间数据隔离；
- UI 越权调用 Tauri；
- 消息洪泛、超大 payload 和无界订阅。

### 19.3 供应链

- 插件/索引/撤销签名失败；
- 发布者密钥变更；
- 同版本内容替换；
- 镜像返回错误包；
- 市场账户接管；
- 恶意依赖和许可证缺失；
- 市场离线和旧撤销缓存。

### 19.4 资源与可靠性

- CPU、内存、磁盘、网络配额；
- 麦克风独占冲突；
- 宿主升级后的协议兼容；
- 多插件贡献同类扩展点；
- 快捷键冲突；
- 全屏/穿透状态下插件展示请求；
- 插件 crash/restart 不污染宿主状态。

### 19.5 官方迁移回归

问答：三个 Provider、联网、取消 A→B、late-result、历史和密钥迁移。
语音：模型安装、唤醒、ASR、TTS、手动录音、问答联动、闲置内存、卸载释放。

## 20. 运维与治理

公开市场上线前必须指定：

- 安全漏洞报告与响应时限；
- 恶意插件/被盗密钥处置；
- 插件下架、撤销与申诉；
- 开发者身份和组织所有权；
- 隐私政策与遥测最小化；
- 审核准则、拒绝理由和重复提交政策；
- 过期宿主版本和不兼容插件生命周期；
- 市场根密钥、在线签名密钥的备份与轮换；
- 安全公告与用户通知渠道。

遥测默认只收集市场/插件运行所需的聚合健康指标，并允许用户控制。不得默认上传对话、语音、文件名、剪贴板或插件私有数据。

## 21. 风险与待专项决策

在阶段 0 必须重新确认：

- 原生 Host 的 Windows 隔离强度与可实现性；
- WASI 运行时选择及体积/性能；
- IPC 编码和大数据流；
- 插件 UI 的 WebView/iframe 边界与 CSP；
- 发布者身份验证强度；
- 市场基础设施托管与成本；
- 离线侧载默认是否允许；
- 依赖插件与能力解析规则；
- 数据迁移和备份保留周期；
- 市场是否永远免费，或未来另设付费项目；
- 对第三方闭源原生插件的审核能力与政策。

这些不是本轮编码 TODO，而是插件专项进入设计前的决策清单。
