# 06-通知系统（Claude/Codex 完成通知）

## 模块做啥（1 行）
`lucky-notify` CLI 直接调用本地 HTTP 端点，主程序分发岛内通知、Windows toast 并保存 SQLite 历史。

## 当前接口

```ts
interface NotifyInput {
  title: string;
  body?: string;
  source: "claude" | "codex" | "custom";
  level: "info" | "success" | "warn" | "error";
  action?: { type: "open_terminal"; cwd: string };
}
```

- `GET 127.0.0.1:9753/health`
- `POST 127.0.0.1:9753/notify`
- Bearer token，兼容 query token。
- 前端事件：`notify://incoming`。

## 当前链路

```text
Claude Code / Codex hook / shell
  → lucky-notify
  → reqwest blocking Client + no_proxy
  → POST 127.0.0.1:9753/notify
  → SQLite notifications
  → notify://incoming
  → 展开岛窗口 / 通知页 / 可选 Windows toast
```

CLI token 来源依次为：

1. `--token`
2. `LUCKY_TOKEN`
3. SQLite `settings['notify:http_token']`

Windows toast 使用 `general:toast` 设置控制。当前 action 仅接受 `open_terminal` 且要求非空 `cwd`。

## 明确不存在

- 没有命名管道或本地 socket 通知通道。
- 没有主进程离线时的待发送队列。
- 没有 `notify/ipc.rs` 或 `dispatcher.rs`；分发逻辑位于现有 notify 模块。
- SQLite schema 由 Rust 启动时内联初始化，没有 `migrations/003_notifications.sql`。

因此，主程序离线或 HTTP 请求失败时 CLI 会直接报错，不承诺后续补发。

## 当前产出


外部 Codex 的 `~/.codex/config.toml` hook 配置仍是合法配置；它与 LuckyIsland 应用配置真源无关。

## 验证要点

- `lucky-notify --title x --source claude` 通过 HTTP 触发通知。
- `curl POST /notify` 带正确 token 成功，错误 token 被拒绝。
- 通知写入 SQLite 历史并发出前端事件。
- `general:toast` 开关控制 Windows toast。
- 代理环境下本地请求不经系统代理。
