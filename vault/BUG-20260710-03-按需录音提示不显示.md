## Bug 摘要
- 实际表现：点击 AI 面板麦克风后，后端日志出现“按需录音线程启动”和“已 emit voice://listening”，但窗口录音期间没有显示“正在聆听”。
- 期望表现：麦克风真正开始采集后立即显示提示；录音成功、失败或超时后提示消失。
- 已排除项：Rust payload 是 boolean，前端 `listen<boolean>` 类型匹配；`app.emit` 为广播；浮层已有相对定位、z-index 和高对比样式，多轮 CSS/payload 猜测未解决。

## 根因
`voice_record_utterance` 是同步 Tauri command。它启动工作线程后在 command 内调用 `result_rx.recv()` 阻塞到录音结束。工作线程虽然成功 `emit("voice://listening", true)`，但 WebView/Tauri command 的事件与返回处理直到同步 command 结束才得到调度；随后前端 `finally` 又立即 `setListening(false)`，所以 true 状态没有可见时段。

当前时序：

```text
前端 invoke
→ 同步 command 阻塞在 recv
→ worker emit true（入队）
→ 录音结束，command 返回
→ listener true 与 finally false 紧邻执行
→ 用户看不到提示
```

## 选定方案（A）
- 将 `voice_record_utterance` 改为 `async fn`。
- 把 cpal 初始化、VAD、ASR 和同步等待整体移入 `tokio::task::spawn_blocking`；async command 本身立即让出执行权。
- 阻塞 helper 在输入流 `play()` 成功后 emit `voice://listening=true`，在所有成功/失败/超时出口 emit false；用 guard 确保 `manual_recording` 与提示状态都能复位。
- 前端按 payload 设置状态，而不是忽略 payload 后一律设 true；可保留超时 timer 作为异常兜底，但 command `finally` 不抢先覆盖正常事件时序。
- 不调整已验证可用的 KWS、VAD 阈值、ASR blank penalty 和 TTS 隔离逻辑。


## 已实施修复
- `voice_record_utterance` 已改为 async Tauri command，并将 cpal/VAD/ASR 阻塞工作整体移入 `tokio::task::spawn_blocking`，使 runtime 能在录音期间向 WebView 投递事件。
- `RecordingGuard` 创建时设置 `manual_recording=true`，Drop 时无条件恢复 false 并 emit `voice://listening=false`；正常返回、错误和 panic unwind 均有测试覆盖。
- 输入流 `play()` 成功后才 emit `voice://listening=true`；启动失败不会伪造正在聆听状态，guard 仍负责清理。
- 前端使用 `listen<boolean>` 并直接消费 `event.payload` 的 true/false；8 秒 timer 只作异常漏发 false 的兜底，manual invoke 的 finally 仅恢复 `recording=false`。
- listener promise 在 React cleanup 后才 resolve 时会立即 unlisten，避免 StrictMode 重挂载泄漏重复 listener。

## 验收标准
- 点击麦克风后，在录音进行中可见“正在聆听”，而不是 invoke 返回后闪现。
- 正常转写、未开口超时、模型/麦克风错误后提示均消失，`manual_recording` 均恢复 false。
- 按需录音期间 KWS 不自触发；结束后 KWS 可继续工作。
- 唤醒路径识别效率维持用户已认可状态，不引入 ASR 重复字回归。
